## 🚨 치명적 버그 목록 (앱이 깨지는 수준)

### 1. JSON ↔ JS 필드명 불일치 — 시즌 캘린더 전체 깨짐

| JS가 기대하는 구조 | 실제 JSON 구조 | 결과 |
|---|---|---|
| `cal.current` | `cal.currentMonth` | ❌ 카드 안 뜸 |
| `cal.current.items[]` | `cal.currentMonth.categories[]` | ❌ 목록 빈 배열 |
| `cal.current.period` | `cal.currentMonth.month` + `year` | ❌ 기간 미표시 |
| `cal.upcoming.items[]` | `cal.upcoming.categories[]` | ❌ 준비 카테고리 안 뜸 |
| `cal.lastYear.items[]` | `cal.lastYear.rankings[]` | ❌ 작년 랭킹 안 뜸 |

**결론: 시즌 캘린더 섹션 전체가 빈 화면.**

### 2. 데일리 업로드 필드명 불일치

| JS 기대 필드 | 실제 JSON 필드 | 결과 |
|---|---|---|
| `u.thumbnail` | `u.imageUrl` | ❌ 썸네일 빈 박스 |
| `u.skuName` | `u.name` | ❌ 상품명 "—" 표시 |
| `day.image` | `day.imageUrl` | ❌ 이미지 없음 |
| `day.margin` | `day.estimatedMargin` | ❌ 마진 미표시 |
| `domesticPrice` 숫자 | ₩ 포맷 없음 | ⚠️ "23000" 그대로 |

### 3. 체크리스트 key/label 구조 불일치

JS 기대: `[{ key: 'photo', label: '상품 사진 촬영 완료' }]`
실제 JSON: `["제품 사진 촬영/수집", "상세페이지 제작"]`

→ `item.key` = undefined, `item.label` = undefined. 체크리스트 이름 안 뜨고 저장 안 됨.

### 4. 미정의 CSS 클래스 18개+

v2에서 추가된 JS가 사용하지만 style.css에 정의되지 않은 클래스:
`.metric-card`, `.metric-icon`, `.metric-value`, `.metric-label`, `.metric-note`,
`.signal-card`, `.signal-title`, `.signal-source`, `.signal-evidence`, `.signal-action`,
`.platform-card`, `.platform-card-head`, `.platform-priority`, `.platform-name`,
`.platform-role`, `.platform-type`, `.platform-watch`, `.category-rail`, `.scale-panel`

## 🔎 근본 원인: 왜 이렇게 됐는가

1. **v1** (첫 세션): CSS + HTML + 9개 JSON + 7개 view JS → 필드명 일치 상태로 커밋
2. **v2** (후속 세션/다른 에이전트): view-cockpit.js, view-sku.js **완전 재작성** + 새 파일 추가 (trend-engine.js, storage-config.js 등)
3. **JSON은 v1 그대로** → v2 JS가 v1 JSON을 읽으려니 필드명 불일치

**핵심**: AI가 이전 컨텍스트를 잃고 호환되지 않는 코드를 덮어쓴 전형적 문제.

## 🛡️ 다음에 비슷한 프로젝트를 할 때 지킬 원칙

### 원칙 1: 데이터-뷰 계약서 먼저 만들어라
- JSON의 모든 필드명을 `docs/data-schema.md`에 정의
- view JS는 이 문서를 "API 문서"처럼 참조
- 필드명 변경 시 schema 문서 먼저 수정 → JS 수정

### 원칙 2: 만들자마자 브라우저에서 열어라
- 3개 파일 만들 때마다 `npx serve .`로 확인
- 콘솔 에러 0개 확인 후 다음 작업
- **"커밋 전 검증" 원칙 반드시 적용**

### 원칙 3: 서브에이전트가 만든 코드는 반드시 통합 검증
- JSON 만드는 에이전트 ↔ JS 만드는 에이전트가 분리되면
- 둘 다 끝난 후 **필드명 매칭 검증 스크립트** 실행
- 예: `grep -oP 'item\.(\w+)' view-cockpit.js` → JSON에 해당 필드 존재 여부 확인

### 원칙 4: 미정의 CSS 클래스 체크
- JS에서 사용하는 `.className`이 style.css에 정의되어 있는지 확인
- 예: `grep -oP 'class="([^"]+)"' view-*.js | sort -u` → style.css에서 검색

### 원칙 5: "완료"라고 하기 전에 체크리스트
- [ ] 브라우저 콘솔 에러 0개
- [ ] 모든 페이지 전환 시 빈 화면 아님
- [ ] 모든 카드에 데이터 표시됨 (— 이나 undefined 없음)
- [ ] 토글/필터/체크리스트 인터랙션 동작
- [ ] LocalStorage 저장/복원 확인
- [ ] 최소 1회 새로고침 후에도 상태 유지

### 원칙 6: 반응형은 처음부터 넣어라
- 미디어쿼리 최소 2개: `max-width: 768px` (태블릿), `max-width: 480px` (모바일)
- 사이드바 → 하단 네비게이션 또는 햄버거 메뉴
- 그리드 → 1열 스택

## ✅ 잘한 점 (다음에도 유지할 것)

1. CSS 변수 기반 디자인 시스템 → 테마 변경 쉬움
2. 사이드바 라우터 패턴 → 페이지 전환 깔끔
3. LocalStorage 상태 관리 → 새로고침 후 유지
4. JSON 데이터 분리 → 코드 수정 없이 데이터 교체 가능
5. 비즈니스 도메인 모델링 → K-뷰티 실무에 맞는 데이터 구조

## ❌ 못한 점 (다음에 반드시 고칠 것)

1. **통합 테스트 0** — 브라우저에서 안 열어봄
2. **데이터-뷰 계약 없음** — 필드명 어긋남
3. **반응형 0** — 모바일 불가
4. **버전 관리 실패** — v2 수정 시 v1 호환성 체크 안 함
5. **"완료"라고 했지만 미완성** — 검증 없이 완료 선언
