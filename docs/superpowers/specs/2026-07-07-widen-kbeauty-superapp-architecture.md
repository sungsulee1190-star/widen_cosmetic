# WIDEN K-Beauty Super App Architecture

작성일: 2026-07-07  
상태: 슈퍼앱 전환 설계 초안  
목적: 현재 SKU 대시보드를 "대표를 위한 K-뷰티 해외판매 운영 슈퍼앱"으로 확장하기 위한 정보구조, DB 구조, 화면 역할, 구현 우선순위를 정의한다.

---

## 1. 제품 정의

WIDEN은 K-뷰티 해외판매 대표를 위해 여러 "운영 직원"이 일하는 것처럼 동작하는 슈퍼앱이다.

단순 대시보드가 아니라, 아래 일을 지속적으로 수행하는 운영 시스템을 목표로 한다.

- 어떤 제품이 뜨는지 찾는다.
- 그 제품이 왜 뜨는지 설명한다.
- 어떤 플랫폼과 매장에서 확인해야 하는지 알려준다.
- 경쟁사가 어떻게 팔고 있는지 분석한다.
- 상세페이지 성공공식을 저장한다.
- 국가별 규제 리스크를 감지한다.
- 브랜드와 성분 지식을 누적한다.
- 이번 주 해야 할 일을 체크리스트로 만든다.

핵심 질문:

```text
이 SKU를 어디서 소싱하고, 어디에 팔고, 어떤 상세페이지/이미지/문구로 팔 것인가?
```

---

## 2. 운영 직원 모델

슈퍼앱은 기능 메뉴가 아니라 "직원 역할"이 협업하는 구조로 설계한다.

| 직원 역할 | 하는 일 | 주요 산출물 |
| --- | --- | --- |
| 트렌드 직원 | 주차별 인기 제품, 성분, 계절성, 스테디셀러 후보를 정리 | 트렌드 뉴스레터 |
| SKU 직원 | 제품별 가격, 마진, 링크, 추천 이력, 상태를 관리 | SKU DB |
| 벤치마킹 직원 | 무신사, Qoo10, Shopee, 올리브영 상세페이지를 분석 | 성공공식 리포트 |
| 규제 직원 | 국가별 금지/주의 성분, 위험 표현, 대체 표현을 관리 | 국가별 규제 DB |
| 브랜드 직원 | 브랜드 역사, 대표 제품, 매출, 직원 수, 유통 채널을 아카이빙 | 브랜드 아카이브 |
| SNS 직원 | 한국, 일본, 중국, 대만 SNS 계정과 콘텐츠를 추적 | SNS/인플루언서 DB |
| 현장 직원 | 명동, 성수, 홍대 매장 방문 기록을 SKU에 연결 | 오프라인 방문 로그 |
| 카피라이팅 직원 | 성공공식을 바탕으로 제목, 썸네일, 상세페이지 초안을 만든다 | 판매페이지 초안 |

## 3. 전체 메뉴 구조

초기 메뉴 구조는 아래처럼 재편한다.

```text
1. 이번주 할일
2. 카테고리/제품
3. 트렌드 뉴스레터
4. 성분/트렌드 DB
5. 벤치마킹샵
6. 경쟁 상세페이지 분석
7. 국가별 규제 DB
8. 브랜드 아카이브
9. 오프라인 소싱 루트
10. 데이터 관리
```

현재 구현된 화면은 이 중 일부의 프로토타입이다. 앞으로는 화면보다 DB 구조를 먼저 안정화한다.

## 4. DB 구조

### 4.1 SKU DB

모든 정보의 중심이다.

필드:

- SKU ID
- 제품명
- 브랜드
- 제품 이미지
- 대분류
- 세부 카테고리
- 주요 성분
- 국내 소싱처
- 국내 소싱 링크
- 해외 판매 플랫폼
- 해외 판매 링크
- 국내 가격
- 해외 가격
- 예상 마진
- 경쟁 페이지 수
- 추천 상태: 즉시 테스트, 관찰, 보류, 규제 확인, 현장 재확인
- 주차별 추천 이력
- 관련 트렌드 뉴스레터
- 관련 성분
- 관련 브랜드
- 관련 경쟁 상세페이지
- 관련 규제 리스크
- 관련 오프라인 방문 기록
- Notion 페이지
- Drive 폴더

### 4.2 주차별 트렌드 뉴스레터 DB

뉴스레터처럼 매주 발행되는 기록이다.

필드:

- 주차
- 제목
- 이번 주 핵심 요약
- 뜨는 SKU
- 뜨는 성분
- 뜨는 브랜드
- 계절성 후보
- 스테디셀러 후보
- 플랫폼별 랭킹 변화
- Qoo10/Shopee 판매 신호
- 올리브영/무신사/다이소 국내 신호
- SNS 신호
- 이번 주 해야 할 일
- 다음 주 관찰 항목
- 관련 SKU
- 관련 브랜드
- 관련 성분

### 4.3 성분/트렌드 DB

성분 설명보다 판매 판단과 마케팅 표현에 초점을 둔다.

필드:

- 성분명
- 대체명/일본어/중국어 표현
- 왜 뜨는가
- 고객이 기대하는 효과
- 대표 SKU
- 관련 브랜드
- 관련 트렌드 주차
- 일본 판매 적합도
- 대만 판매 적합도
- 쓸 수 있는 표현
- 위험한 표현
- 대체 표현
- 관련 규제
- 관련 콘텐츠/SNS 링크
- 누적 관찰 메모

### 4.4 국가별 규제 DB

고정 안내 페이지가 아니라 계속 쌓이는 리스크 DB로 운영한다.

필드:

- 국가
- 규제 항목
- 관련 성분
- 관련 표현
- 위험도: 낮음, 중간, 높음
- 영향 받는 SKU
- 문제 문구
- 대체 문구
- 해결 액션
- 출처 링크
- 마지막 확인일
- 담당 상태: 확인 필요, 반영 완료, 보류

예시:

```text
SKU: 넘버즈인 글루타치온 토너
국가: 일본
위험: 미백 표현 주의
문제 문구: 美白, シミ改善
대체 문구: 透明感, ツヤ, 明るい印象
해결 액션: 상세페이지 문구 수정
```

### 4.5 벤치마킹샵 DB

플랫폼별로 계속 봐야 할 샵을 누적한다.

필드:

- 샵명
- 플랫폼: Qoo10 JP, Shopee TW, 무신사, 올리브영, 다이소, Amazon JP, Rakuten JP
- 유형: 공식샵, K-뷰티 전문샵, 종합 편집샵, 리셀러, 오프라인 매장
- URL
- 강한 카테고리
- 주요 SKU
- 제목 전략
- 이미지 전략
- 쿠폰/세트 전략
- 상세페이지 특징
- 리뷰 활용 방식
- 벤치마킹 포인트
- 관련 경쟁 상세페이지
- 마지막 관찰일

### 4.6 SNS/인플루언서 DB

뷰티 제품을 많이 소개하는 한국, 일본, 중국, 대만 계정을 누적한다.

필드:

- 계정명
- 플랫폼: TikTok, Instagram, YouTube, Xiaohongshu, 블로그, 커뮤니티
- 국가/언어
- URL
- 주요 카테고리
- 자주 다루는 브랜드
- 자주 다루는 SKU
- 강한 콘텐츠 형식: 리뷰, 비교, 루틴, 언박싱, before/after
- 참고할 톤앤매너
- 반복 고객 언어
- 관련 SKU
- 마지막 관찰일

### 4.7 경쟁 상세페이지 분석 DB

무신사, Qoo10, Shopee, 올리브영 등의 상세페이지를 해부해 성공공식으로 저장한다.

필드:

- 분석 ID
- URL
- 플랫폼
- 브랜드
- SKU
- 판매자/샵
- 첫 화면 이미지
- 전체 톤앤매너
- 색감
- 사진 스타일
- 폰트 느낌
- 상세페이지 섹션 순서
- 첫 문장/후킹 문구
- 주요 효능 표현
- 성분 설명 방식
- 리뷰/근거 활용 방식
- 가격/쿠폰/세트 구성
- CTA 문구
- 따라 할 점
- 변형해서 쓸 점
- 피해야 할 점
- 성공공식 요약
- 관련 SKU
- 관련 규제 리스크
- 캡처 이미지 Drive 링크

성공공식 예시:

```text
일본 Qoo10 PDRN 세럼 상세페이지
- 첫 화면: 성분명 + 피부고민 + 랭킹 문구
- 색감: 흰색/하늘색/메디컬 톤
- 사진: 제품 단독 + 제형 클로즈업
- 문구: 毛穴, ハリ, ツヤ, 低刺激
- CTA: 메가와리 쿠폰 + 2개 세트
```

### 4.8 브랜드 아카이브 DB

브랜드를 장기적으로 이해하기 위한 지식 DB다.

필드:

- 브랜드명
- 공식 홈페이지
- 국가
- 브랜드 유형: 대형, 중소형, 인디, 급상승
- 설립연도
- 창업자/운영사
- 브랜드 역사
- 대표 제품
- 대표 성분
- 매출
- 직원 수
- 매장/입점 채널
- 주요 판매 국가
- 주요 고객층
- 유통 통제 강도
- 소싱 가능성
- 관련 SKU
- 관련 경쟁 페이지
- 관련 SNS 계정
- 참고 기사/출처
- 마지막 업데이트일

### 4.9 오프라인 방문 로그 DB

명동, 성수, 홍대 중심으로 운영한다.

필드:

- 방문일
- 권역
- 매장명
- 매장 유형
- 확인 SKU
- 가격
- 재고
- 진열 위치
- 세일/기획전
- 관광객 반응
- 사진 Drive 링크

## 5. 화면별 역할

### 5.1 이번주 할일

대표가 매일 여는 화면이다.

보여줄 것:

- 오늘 해야 할 액션
- 이번 주 트렌드 뉴스레터 요약
- 새로 봐야 할 SKU
- 경쟁 상세페이지 분석 필요 항목
- 규제 확인 필요 SKU
- 방문할 매장
- Notion/Drive 업데이트 필요 항목

### 5.2 카테고리/제품

제품 탐색과 SKU 상태 관리 화면이다.

보여줄 것:

- 무신사식 대분류 가로 탭
- 주차별 추천 필터
- 플랫폼별 랭킹/가격 필터
- 인기 판단 기준 슬라이더
- 제품 사진과 링크
- 제품별 장단점
- 주차별 추천 이력
- 상세 분석 패널

### 5.3 트렌드 뉴스레터

주차별 리포트 화면이다.

보여줄 것:

- 2026-W28 같은 주차 선택
- 뜨는 제품
- 뜨는 성분
- 계절성 제품
- 스테디셀러 후보
- 플랫폼별 변화
- 다음 주 액션

### 5.4 성분/트렌드 DB

성분을 판매 문구와 규제 리스크까지 연결한다.

### 5.5 벤치마킹샵

플랫폼별로 볼 샵과 계정을 관리한다.

### 5.6 경쟁 상세페이지 분석

상세페이지 성공공식을 만드는 핵심 화면이다.

상세페이지 URL을 기준으로:

- 화면 캡처
- 톤앤매너 분석
- 색감/사진/폰트/배치 분석
- 상세페이지 구조 분석
- 따라 할 점과 변형할 점 정리
- 성공공식 저장

### 5.7 국가별 규제 DB

국가별 위키가 아니라 "내 SKU에 적용되는 리스크" 중심으로 보여준다.

### 5.8 브랜드 아카이브

브랜드의 장기 지식과 소싱 판단 자료를 쌓는다.

### 5.9 데이터 관리

Notion, Google Drive, MD 리포트, 캡처 파일 구조를 관리한다.

## 7. 구현 우선순위

### 1차: 정보구조와 DB 스키마

- 이 문서를 기준으로 DB 필드를 확정한다.
- Notion DB 생성 또는 템플릿을 만든다.
- 현재 웹 대시보드 메뉴를 슈퍼앱 구조로 맞춘다.

### 2차: 경쟁 상세페이지 분석

- 무신사 상세페이지 분석 템플릿
- Qoo10 상세페이지 분석 템플릿
- 성공공식 저장 화면
- Drive 캡처 링크 연결

### 3차: 트렌드 뉴스레터

- 주차별 뉴스레터 화면
- 제품/성분/브랜드/계절성/스테디셀러 연결
- 다음 주 액션 자동 생성

### 4차: 브랜드 아카이브

- 브랜드 홈페이지 기반 정보 수집
- 대표 제품, 매출, 직원 수, 매장, 유통 채널 기록
- 대형/중소/인디/급상승 분류

### 5차: 규제 리스크 연결

- 국가별 규제 DB와 SKU DB 연결
- 위험 표현 감지
- 대체 표현 제안

## 9. 결론

WIDEN은 예쁜 대시보드가 아니라 대표를 위한 운영 직원들의 슈퍼앱이 되어야 한다.

가장 중요한 기준은 다음이다.

- 모든 정보는 SKU와 연결된다.
- 매주 뉴스레터처럼 트렌드가 쌓인다.
- 경쟁 상세페이지는 성공공식으로 저장된다.
- 국가별 규제는 내 SKU의 위험과 해결 액션으로 연결된다.
- 브랜드, 성분, SNS, 오프라인 방문 기록이 장기 아카이브가 된다.
- 웹 대시보드는 모든 자료를 다 담지 않고, 오늘의 판단과 액션을 보여준다.
