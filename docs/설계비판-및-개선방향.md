# WIDEN K-Beauty Dashboard 설계 비판 및 개선 방향

작성일: 2026-07-22
문서 상태: 교차검증 완료 및 합의안 (Final)

## 1. 문서 목적

이 문서는 현재 대시보드의 설계와 운영 방식을 비판적으로 검토하고, 앞으로 상품 발굴·평가·상세페이지 제작을 어떤 구조로 바꿀지 정의한다.

이 프로젝트의 핵심 문제는 화면이 부족한 것이 아니라, 상품 데이터와 판단 근거가 갱신되지 않는다는 점이다. 따라서 이후 기능 개발은 새로운 화면을 계속 추가하는 방식보다 다음 흐름을 먼저 완성해야 한다.

`후보 수집 → 출처 기록 → 교차검증 → 점수화 → 선정/보류 → 주간 이력 저장`

## 2. 현재 설계에 대한 비판

### 2.1 고정 샘플 데이터가 실제 운영 데이터처럼 보인다

특정 상품이 여러 화면의 대표 상품으로 반복 노출되면 사용자는 해당 상품이 최신 트렌드로 선정된 것인지, 단순한 예시인지 구분할 수 없다. 상품 데이터에 기준일·출처·선정 이유가 없으면 대시보드는 정보 도구가 아니라 정적인 소개 페이지가 된다.

개선 원칙:

- 대표 상품을 코드에 고정하지 않는다.
- 모든 후보에 `수집일`, `기준 주차`, `국가`, `출처 URL`, `선정 상태`를 저장한다.
- 화면에는 “왜 이 상품이 올라왔는가”를 반드시 표시한다.

### 2.2 트렌드와 판매 가능성이 한 점수로 섞여 있다

검색량이나 소셜 화제성만 높으면 단기 유행일 수 있다. 반대로 판매량이 오래 유지되어도 콘텐츠로 확장하기 어려울 수 있다. 두 요소를 하나의 임의 점수로 합치면 사용자가 결과를 신뢰하기 어렵다.

개선 원칙:

- `판매 가능성`과 `콘텐츠 가능성`을 별도 점수로 관리한다.
- 계절성은 감점 항목으로 명시한다.
- 점수와 함께 근거가 부족한 항목을 표시한다.

### 2.3 출처의 종류와 중요도가 정의되어 있지 않다

올리브영, Qoo10, 현지 경쟁 셀러, TikTok, Google Trends는 서로 다른 질문에 답한다. 이들을 같은 수준의 근거로 취급하면 한국 내 인기와 목표 국가의 실제 수요를 혼동하게 된다.

개선 원칙:

- 목표 국가의 판매·검색 데이터를 1차 수요 근거로 둔다.
- 올리브영과 다이소는 공급 후보와 가격대 확인에 사용한다.
- 소셜 채널은 확산 가능성 확인에 사용한다.
- 한 출처만으로 “잘 팔린다”고 결론 내리지 않는다.

### 2.4 데이터가 비어도 실패 원인이 보이지 않는다

링크 오류, 외부 이미지 차단, JSON 로딩 실패가 빈 화면으로 보이면 사용자는 데이터가 없는 것인지 기능이 고장 난 것인지 알 수 없다.

개선 원칙:

- 출처별 성공·실패 상태를 저장한다.
- 빈 화면 대신 “확인 실패”, “수동 입력 필요”, “최근 갱신일”을 표시한다.
- 외부 링크와 이미지는 오류 대체 화면을 제공한다.

### 2.5 브라우저별 저장과 운영 기준이 분리되어 있다

브라우저 로컬 저장만 사용하면 Chrome과 Edge, 다른 컴퓨터가 서로 다른 데이터를 보게 된다. 또한 점수 기준을 문서로만 관리하면 시간이 지나면서 평가 방식이 달라진다.

개선 원칙:

- 운영 데이터는 공용 저장소를 기준으로 한다.
- 점수표는 버전과 적용 주차를 함께 저장한다.
- 모든 수정은 변경자·변경일·변경 이유를 남긴다.

## 3. 개선된 운영 모델

### 3.1 데이터 계층

1. `source`: Qoo10, 올리브영, Rakuten, 현지 셀러, Google Trends, TikTok 등 출처
2. `candidate`: 검토할 상품 후보
3. `evidence`: 후보를 뒷받침하는 순위·가격·리뷰·검색·콘텐츠 근거
4. `scorecard`: 고정된 평가 기준과 점수
5. `decision`: 선정, 보류, 탈락, 재검토 상태
6. `weekly_snapshot`: 주차별 점수와 순위 변화
7. `content_brief`: 키워드, 포지셔닝, 상세페이지, 이미지 제작 지시서

### 3.2 상품 상태

- `수집`: 후보로 등록되었지만 검증 전
- `검증 중`: 최소 두 종류의 출처를 확인하는 단계
- `추천`: 기준 점수를 통과한 상품
- `보류`: 근거 부족, 공급 불안정, 규제 검토 필요
- `탈락`: 기준 미달 또는 위험 요인 존재
- `추적`: 이번 주 선정하지 않았지만 다음 주 재확인할 상품

### 3.3 가장 중요한 화면

첫 화면은 고정된 상품 카드가 아니라 “이번 주 의사결정 보드”가 되어야 한다.

- 이번 주 새 후보
- 지난주 대비 상승·하락
- 근거가 두 개 이상 확인된 후보
- 점수는 높지만 공급 확인이 안 된 후보
- 점수는 낮아도 재검토할 후보
- 선정·보류·탈락 사유

## 4. 개선 우선순위

### 1단계: 기준과 저장 구조 고정
평가 기준표, 출처 유형, 상품 상태, 주차별 이력을 먼저 확정한다.

### 2단계: 수동 큐레이션 완성
사용자가 후보와 링크를 직접 입력하면 시스템이 근거와 점수를 정리하고, 상위 후보를 보여준다.

### 3단계: 교차검증 보조
후보별로 목표 국가 수요, 한국 공급, 검색 추세, 소셜 가능성, 위험 요소를 한 화면에서 비교한다.

### 4단계: 콘텐츠 패키지 생성
선정된 후보에서 키워드, 상품 포지셔닝, 상세페이지 구성, 촬영 목록, 이미지 생성 지시서를 만든다.

### 5단계: 자동 수집 확대
충분한 주차별 이력이 쌓인 뒤에만 자동 수집·자동 점수화를 추가한다. 자동화 전에도 사람이 근거를 승인하는 구조를 유지한다.

## 5. 결론

이 대시보드는 “트렌디한 상품을 보여주는 화면”이 아니라 “매주 같은 기준으로 상품을 선택하고, 그 판단을 다시 검증하는 운영 시스템”이어야 한다. 기준표와 이력 저장이 먼저 완성되지 않으면 자동화는 고정 샘플을 더 빠르게 반복하는 기능이 될 뿐이다.

## 6. 교차검증 최종 결정사항 및 개발 실행 방향 (Antigravity AI 파트너 합의 - 2026-07-22)

GPT 비판 및 개선 방향성에 100% 동의하며, 피터(사용자)와의 상의 하에 다음과 같이 개발 실행 방침을 최종 확정함:

1. **데이터 스키마(계약서) 코드화**: `data/scorecard-v1.json` 및 후보/근거 데이터 스키마를 정립하여 JSON 데이터와 JS 인터페이스 간 불일치(undefined 버그)를 근본 차단함.
2. **콕핏 화면의 '주간 의사결정 보드' 재구성**: `js/view-cockpit.js`를 개편하여 주간 후보(일본 5개, 대만 5개)의 9항목 가중치 점수 계산기, 근거 URL 노출, `추천/보류/탈락` 1초 판단 버튼 및 판정 사유 입력 UI 탑재.
3. **데이터 방어벽(Fallback) 구축**: 외부 링크 파손 또는 이미지 핫링킹 차단 시 빈 화면이 뜨지 않고 '수동 확인 필요' 태그가 뜨도록 예외 처리 강화.
4. **로컬 영속화 및 주간 스냅샷**: `js/data-store.js` 및 LocalStorage를 연동하여 주차별 평가 결과와 판단 근거가 보존되도록 구현.
